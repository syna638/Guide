Project Overview

Basic What is Smart Civic Connect?

Smart Civic Connect is a full-stack web application that enables citizens to report civic issues like potholes, broken streetlights, and sanitation problems to their local government.

Key features include:

  • Photo and GPS location submission
  • Real-time status tracking (Pending ‚Üí In Progress ‚Üí Resolved)
  • Admin dashboard for government officials to manage and resolve complaints
  • PWA support for installation on any device

It bridges the gap between citizens and government by digitizing the complaint process.

Basic What problem does this project solve?

Traditional civic complaint systems are frustrating: long phone waits, lost paperwork, and no visibility into complaint status.

SCC solves this by:

  • Instant submission: Report issues in seconds from your phone
  • Transparency: Track exactly where your complaint is in the process
  • Evidence: Photos and GPS make issues clear and verifiable
  • Efficiency: Admins can prioritize, filter, and resolve issues faster
  • Accountability: Digital trail ensures complaints aren't lost
Basic What technologies did you use and why?

Next.js 15 (React): Full-stack React framework with server-side rendering, API routes, and optimized performance. Chose it for the unified frontend/backend experience.

Clerk: Premium authentication service. Handles all auth complexity (social logins, email verification, session management) so I could focus on core features.

Supabase: Open-source Firebase alternative. Provides PostgreSQL database with Row Level Security, perfect for multi-tenant data where users should only see their own complaints.

Tailwind CSS: Utility-first CSS framework for rapid, consistent UI development.

TypeScript: Type safety catches bugs before runtime and improves developer experience.

Vercel: Purpose-built for Next.js with automatic deployments, edge functions, and global CDN.

Frontend & React

Basic What is the difference between client and server components in Next.js?

Server Components (default in Next.js 13+):

  • Rendered on the server
  • Can directly fetch data from databases
  • No JavaScript sent to client (smaller bundle)
  • Cannot use hooks like useState, useEffect

Client Components (marked with 'use client'):

  • Rendered in the browser
  • Can use React hooks and browser APIs
  • Required for interactivity (forms, buttons, animations)
  • JavaScript included in client bundle

In SCC, I use Server Components for data fetching (complaint lists) and Client Components for interactive elements (forms, modals, location picker).

Intermediate How do you handle form submissions in Next.js 15?

I use Server Actions, a Next.js feature that lets you define server-side functions that can be called directly from forms.

// lib/actions/complaints.ts
'use server'

export async function createComplaint(formData: FormData) {
  const { userId } = await auth()  // Get authenticated user
  if (!userId) throw new Error('Unauthorized')
  
  // Validate and process data
  const title = formData.get('title')
  
  // Insert into database
  await supabase.from('complaints').insert({...})
  
  // Revalidate cache
  revalidatePath('/complaints')
}

Benefits:

  • No need for separate API routes
  • Automatic form handling with progressive enhancement
  • Type-safe with TypeScript
  • Works even if JavaScript fails (graceful degradation)
Intermediate How did you implement the location picker feature?

I used Leaflet and React-Leaflet for interactive maps:

  • Map displays with OpenStreetMap tiles (free)
  • User can click to place a marker or use their GPS location
  • Coordinates (lat/lng) are captured and stored with the complaint
  • Location is displayed as a map preview in complaint details

The component is a Client Component because it requires browser APIs (geolocation, DOM manipulation).

// Uses browser geolocation API
navigator.geolocation.getCurrentPosition(
  (position) => {
    setLocation({
      lat: position.coords.latitude,
      lng: position.coords.longitude
    })
  }
)
Basic How did you implement dark mode?

I used the next-themes library which:

  • Persists theme preference in localStorage
  • Respects system preference by default
  • Prevents flash of wrong theme on load
  • Adds a class to the HTML element that Tailwind uses for dark variants

Tailwind's dark: prefix applies dark mode styles:

<div className="bg-white dark:bg-gray-900 text-black dark:text-white">

Backend & APIs

Intermediate How do you handle file uploads?

Image uploads go to Supabase Storage:

  1. Client selects image from camera or gallery
  2. Image is compressed client-side for faster upload
  3. Uploaded to Supabase Storage bucket complaint-images
  4. Public URL is returned and stored with the complaint
const { data, error } = await supabase.storage
  .from('complaint-images')
  .upload(`${userId}/${fileName}`, file)

// Get public URL
const { publicUrl } = supabase.storage
  .from('complaint-images')
  .getPublicUrl(data.path)

Security: Storage policies ensure only authenticated users can upload, and files are organized by user ID.

Advanced Explain how you handle authentication in middleware.

Our middleware.ts handles dual authentication systems:

1. Route Classification:

const isAdminRoute = createRouteMatcher(["/admin(.*)"])
const isCitizenRoute = createRouteMatcher(["/home(.*)", "/complaints(.*)"])
const isPublicRoute = createRouteMatcher(["/", "/sign-in(.*)"])

2. Admin Routes (/admin/*):

  • Read admin-session cookie from request
  • Verify JWT using jose library (Edge-compatible)
  • If valid, allow access; if not, redirect to /admin/login

3. Citizen Routes (/home, /complaints):

  • Use Clerk's auth() to check session
  • If no userId, call redirectToSignIn()

4. Supabase Session:

  • Refresh Supabase session tokens at the start of every request
  • Ensures database queries use fresh credentials

This runs at the Edge (before the page even loads) for maximum security and performance.

Intermediate Why did you use two separate authentication systems?

Security through separation:

  • Admin credentials are completely isolated from citizen accounts
  • Even if a citizen account is compromised, admin access is safe
  • Admin auth uses simpler, more controlled username/password (no social login attack surface)

Different requirements:

  • Citizens: Need easy signup, social logins, password reset ‚Üí Clerk excels at this
  • Admins: Few users, controlled credentials, simpler is better ‚Üí Custom JWT

Cost efficiency:

  • Clerk charges per Monthly Active User
  • Keeping admin auth separate means admins don't count against Clerk limits

Database & Security

Intermediate What is Row Level Security (RLS) and how do you use it?

Row Level Security is a PostgreSQL feature that controls which rows each user can access at the database level.

Why it matters: Even if someone bypasses your app code, the database itself enforces access rules.

Example policy:

-- Users can only SELECT their own complaints
CREATE POLICY "Users can view own complaints"
  ON complaints
  FOR SELECT
  USING (user_id = current_setting('request.jwt.claims')::json->>'sub');

In SCC:

  • Citizens can only read/write their own complaints
  • Admin operations use the service_role key which bypasses RLS
  • Categories are readable by everyone (no user restriction)
Basic How is the database structured?

We have four main tables:

complaints: Core table with title, description, category_id (FK), user_id, status, image_url, location data, admin_response, timestamps.

categories: Lookup table with name, description, icon. Seeded with Roads, Water, Electricity, Sanitation, etc.

profiles: Extended user info linked to Clerk user IDs.

notifications: Status update alerts with read/unread state.

Relationships:

  • complaints.category_id ‚Üí categories.id (many-to-one)
  • complaints.user_id ‚Üí Clerk user ID (string reference)
Advanced How do you handle the different API keys for Supabase?

Supabase provides two key types:

1. anon/public key (NEXT_PUBLIC_SUPABASE_ANON_KEY):

  • Safe to expose in client-side code
  • All queries are subject to RLS policies
  • Used for citizen operations

2. service_role key (SUPABASE_SERVICE_ROLE_KEY):

  • NEVER expose to client (server-only)
  • Bypasses all RLS policies
  • Used for admin operations and server-side data fetching

Implementation:

// Client operations - uses anon key, respects RLS
const supabaseClient = createBrowserClient(url, anonKey)

// Server operations - uses service role for full access
const supabaseAdmin = createServiceClient(url, serviceKey)

Architecture & Design Decisions

Advanced What would you change if you had to scale this to millions of users?

Database:

  • Add indexes on frequently queried columns (status, created_at, user_id)
  • Implement database partitioning by date or region
  • Consider read replicas for heavy read operations

Caching:

  • Add Redis for session caching and frequently accessed data
  • Implement ISR (Incremental Static Regeneration) for dashboard pages
  • Use CDN for complaint images

Architecture:

  • Separate read/write APIs for different scaling needs
  • Implement queue system for image processing
  • Consider microservices if different complaint types have different processing needs

Monitoring:

  • Add proper observability (logs, metrics, traces)
  • Implement rate limiting per user
  • Set up alerting for performance degradation
Intermediate How do you handle errors in the application?

Client-side:

  • React Error Boundaries catch component errors
  • Form validation with Zod before submission
  • Toast notifications for user-friendly error messages

Server-side:

  • Try-catch in all Server Actions
  • Custom error responses with appropriate status codes
  • Structured error logging for debugging

Middleware:

  • Global catch-all error handler
  • Graceful degradation (if Supabase session refresh fails, continue with request)
  • Different error handling for dev vs production (detailed logs in dev only)
// error.tsx - Next.js error boundary
export default function Error({ error, reset }) {
  return (
    <div>
      <h2>Something went wrong!</h2>
      <button onClick={reset}>Try again</button>
    </div>
  )
}
Intermediate What is a PWA and why did you implement it?

A Progressive Web App is a website that can be installed on devices and work like a native app.

SCC as a PWA provides:

  • Installable: Users can add to home screen, no app store needed
  • Offline support: Cached pages work without internet
  • Native feel: Full-screen mode, app icon, splash screen
  • Cross-platform: Same codebase works on phones, tablets, desktops

Implementation:

  • Used @ducanh2912/next-pwa package
  • Created manifest.json with app metadata
  • Generated icons in all required sizes
  • Service worker auto-generated during build

Benefits for civic apps: Users in low-connectivity areas can still view their complaints and compose new ones for later submission.

Behavioral & Situational

Basic What was the most challenging part of this project?

The dual authentication system was the biggest challenge.

The problem: Next.js middleware runs at the Edge, but many auth libraries use Node.js-specific APIs that don't work there.

What I learned:

  • Edge Runtime limitations and how to work around them
  • How to use jose library for Edge-compatible JWT verification
  • The importance of reading library documentation carefully
  • Debugging auth issues by breaking down the flow step by step

Solution: Created separate auth utilities specifically for middleware that don't use any server-only APIs.

Basic If you could start over, what would you do differently?

Start with better planning:

  • Document the database schema before coding
  • Plan the auth strategy earlier (the dual-auth complexity emerged over time)

Testing:

  • Add unit tests for Server Actions from the beginning
  • Set up E2E tests with Playwright earlier

Code organization:

  • More consistent error handling patterns from day one
  • Better type definitions for API responses

That said, the iterative approach helped me learn and adapt as requirements became clearer.

Basic How would you explain this project to a non-technical person?

"Have you ever seen a pothole and wished you could just take a photo and send it to the city? That's exactly what Smart Civic Connect does.

You open the app, take a photo of the problem, and it automatically knows your location. You add a quick description and submit.

On the other side, city officials see all these reports on a dashboard. They can mark things as 'we're working on it' or 'fixed', and you get updates on your phone.

It's like tracking a package, but for neighborhood problems. Everyone can see their issues are being addressed, and the city can respond faster because everything's organized in one place."
